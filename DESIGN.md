# DESIGN.md — yahoo-mail-mcp

## 1. Goals and non-goals

**Goal.** Focused MCP for triaging a large, spammy Yahoo Mail mailbox. Read, search, move, delete, mark-read, bulk-purge-from-sender. Multi-account from day 1.

**Non-goals (deliberate, do not slowly creep into these):**
- Not a general-purpose IMAP MCP. Yahoo quirks baked in; other providers may work but aren't the target.
- Not a mail client. No send, no compose, no reply.
- Not a corpus indexer. Search returns *server-side* hits, not a local cache.
- Not a thread reconstructor. Yahoo's IMAP threading is unreliable; we don't try.
- Not push / IDLE. Yahoo drops IDLE at ~9 minutes; polling is more reliable for triage.

## 2. Why IMAP (and not something else)

Researched 2026-05-28. **There is no better programmatic path for third-party Yahoo Mail access in 2026:**

- Yahoo's developer OAuth2 Mail API was retired to new third-party app registrations years ago. The current `senders.yahooinc.com` developer program is a gated vendor application that hands you the same IMAP+OAuth2 protocol behind paperwork.
- Microsoft Graph "Connected Accounts" (which used to let you add Yahoo to Outlook.com) was retired in 2018-2019.
- JMAP, EWS, ActiveSync: not supported by Yahoo.
- Commercial unified-email API gateways (Nylas, Aurinko, Unipile) all ride IMAP under the hood for Yahoo. For a single mailbox you already own with an app password, they're $180+/year for zero new capability.
- AOL (Yahoo's sister service) has no public Mail API.
- Browser automation against `mail.yahoo.com` is fragile, slow at 100k+, and ToS-marginal.

**IMAP it is.** App-password auth, port 993, TLS required.

## 3. Authentication

**App passwords only.** Generated by the user at Yahoo Account Security → Generate app password. 16-character lowercase strings, no spaces in the value (Yahoo's UI presents them as 4 groups of 4 separated by spaces — strip).

OAuth2 is not available to us (see §2). We don't try.

## 4. Tool surface (v0 MVP)

All tools take a `host` parameter (the account nickname from config). No defaults — the LLM must be explicit about which account.

| Tool | Args | Returns | Notes |
|---|---|---|---|
| `list_accounts` | — | `[{name, address, default_folder}]` | Reads config, no IMAP connect |
| `list_folders` | `host` | `[{name, normalized, count, unseen}]` | LIST + STATUS; cached per session |
| `search` | `host`, `folder`, `criteria{from?, to?, subject?, body?, since?, before?, seen?, unseen?, larger?, smaller?}`, `limit=100`, `offset=0` | `{total, hits: [{uid, from, subject, date, size, flags}]}` | IMAP SEARCH; total is capped/estimated for very large hits |
| `fetch_message` | `host`, `folder`, `uid`, `fields="headers"\|"body"\|"full"` | `{uid, headers, body_text?, body_html?, attachments_meta?}` | One message |
| `fetch_messages_bulk` | `host`, `folder`, `uids[]≤100`, `fields="headers"` | `[message...]` | Bulk fetch; capped at 100 |
| `move_message` | `host`, `folder`, `uid`, `target_folder` | `{moved: true, new_uid}` | UID MOVE if supported, else COPY+EXPUNGE |
| `delete_message` | `host`, `folder`, `uid`, `expunge=true` | `{deleted: true}` | Flag + expunge by default |
| `mark_read` | `host`, `folder`, `uid`, `read=true` | `{flags}` | Toggle \\Seen |
| `bulk_purge_from` | `host`, `folder`, `from_address`, `confirm_count` | `{purged_count}` | Phase 2. Requires `confirm_count` to match SEARCH count exactly — refuses if mailbox grew between SEARCH and EXPUNGE |

## 5. Yahoo IMAP quirks (must handle)

| Quirk | Mitigation |
|---|---|
| `\Bulk Mail` / `\Junk` / `\Trash` folder naming inconsistent across older accounts | `list_folders` normalizes to canonical lowercase: `inbox`, `sent`, `drafts`, `trash`, `spam`, `archive`. Original name kept as `name`, canonical as `normalized` |
| `SEARCH ALL` on 100k+ mailbox can timeout or partial-return | Paginate UID space: `SEARCH UID 1:1000` chunks, accumulate, max 100k UIDs before refusing with "too broad" |
| `LIST "" "*"` slow on accounts with many folders; sometimes times out | Retry with exponential backoff (3 attempts, 1s/3s/9s) |
| Yahoo drops IDLE at ~9 minutes | We don't use IDLE. All operations poll-on-demand. |
| Yahoo evicts idle connections at ~10 minutes | Connection pool with 8-minute proactive eviction. Reconnect-on-reuse. |
| Some legacy messages have malformed quoted-printable encoding | `imap_tools` handles most; fall back to raw bytes + manual decode with `errors="replace"` |
| Attachment content-type sometimes mislabeled | Trust `Content-Disposition: attachment` over `Content-Type` for is-attachment determination |
| Spam folder UIDs may renumber after Yahoo's automatic purge | UIDVALIDITY check on every connect; invalidate cached UIDs if changed |
| Yahoo auto-expunges on `STORE \\Deleted` (no explicit EXPUNGE needed) | `delete_message(expunge=False)` is silently a no-op on Yahoo; documented in the tool docstring. The returned `expunged` field reflects the caller's intent, not Yahoo's actual behavior |

## 6. Configuration

**Default location:** `~/.config/yahoo-mail-mcp/config.toml`
**Override:** `YAHOO_MAIL_MCP_CONFIG=/absolute/path`

```toml
[defaults]
host = "imap.mail.yahoo.com"
port = 993
use_tls = true
connect_timeout = 10
read_timeout = 30

[accounts.primary]
address = "you@yahoo.com"
# password sourced from YAHOO_MAIL_MCP_PRIMARY_PASSWORD env var (preferred)
# password = "..."  # only for testing; never commit

[accounts.work]
address = "you@yahoo-inc.example"
# password sourced from YAHOO_MAIL_MCP_WORK_PASSWORD
```

**Precedence (highest to lowest):**
1. Per-call tool argument (not currently exposed; future)
2. `YAHOO_MAIL_MCP_<ACCOUNT>_PASSWORD` env var
3. Config file `password` field
4. Error: missing credential

## 7. Connection lifecycle

- One IMAP connection per account, lazily created
- Connection pool with proactive 8-minute eviction (under Yahoo's 10-min idle-timeout)
- `auth_login(username, app_password)` on connect
- `SELECT` folder before any per-folder operation; track current folder to skip redundant SELECTs
- `LOGOUT` on shutdown

## 8. Error model

Every tool returns either:
- Success: typed result per the schema
- Error: `{error: "code", message: "human readable", retryable: bool}`

Error codes: `auth_failed`, `connection_failed`, `folder_not_found`, `message_not_found`, `uid_invalidated`, `search_too_broad`, `rate_limited`, `unknown`. The LLM gets enough info to retry or escalate.

## 9. Testing

- **Unit:** mocked IMAP responses via `imaplib`-shaped fixtures. Cover error paths, folder normalization, UID pagination, UIDVALIDITY invalidation. Target ≥90% line coverage.
- **Live integration:** behind `pytest -m live`, gated on `YAHOO_MAIL_MCP_LIVE=1`. Runs against the configured `primary` account. Read-only by default; opt-in `--allow-writes` flag for move/delete tests against a designated `_mcp-test/` folder.

## 10. Open questions

- **Bulk delete confirmation UX.** `bulk_purge_from` requires `confirm_count` to match a prior SEARCH exactly. Should we also support a dry-run mode that returns the UIDs without deleting? Probably yes; deferred to Phase 2.
- **Trash vs delete.** Some accounts have a `\Trash` folder that's the standard target for delete; others have `Deleted Items` or nothing. `delete_message` defaults to flag+expunge (gone, no trash). Should there be a `to_trash=true` option that moves instead? Probably yes; deferred.
- **Multi-account parallelism.** v0 serializes per-account. If user wants to run the same triage across all accounts at once, do we parallelize internally or require the LLM to fan out? Punt to Phase 2.

## 11. Release plan

- **v0.0.x (Alpha):** skeleton + config + auth + `list_accounts` / `list_folders` (no live IMAP)
- **v0.1.x (Alpha):** + `search` / `fetch_message` / `fetch_messages_bulk` (read-only complete)
- **v0.2.x (Beta):** + `move_message` / `delete_message` / `mark_read`
- **v0.3.x (Beta):** + `bulk_purge_from` + Yahoo quirk hardening + live integration test suite
- **v1.0.0:** trouble-free use against the live 100k+ account for ≥3 weeks; promote `Development Status` to `5 - Production/Stable`.

## 12. Out-of-scope but considered

- **OAuth2 via senders.yahooinc.com gated program.** Requires vendor application + security review. Would replace app-password auth with OAuth2 token refresh. Same IMAP underneath. Skipped: pointless paperwork for an in-house tool.
- **Unified email API gateway (Nylas/Aurinko).** $180+/yr for IMAP passthrough. Skipped: zero new capability over direct IMAP.
- **Local cache / corpus index.** Belongs in a different MCP layer if needed at all. The Outlook archive on CS2 already handles corpus reasoning for the catorcini.com mail. Yahoo is a triage-and-purge target, not an archive target.
